Sentry OHS Bug & Consistency Report

Executive Summary

Review focused on security, tenant isolation, authorization consistency, and runtime reliability across PHP pages and APIs.

Total confirmed findings: 7 (1 Critical, 3 High, 3 Medium).

Confirmed Findings

Critical Incident review update is blocked by CSRF mismatch

Issue: The incident review API requires CSRF validation, but the management page posts JSON without a CSRF token/header, causing update failures.

Risk: Managers cannot reliably save investigation/classification changes (functional outage).

Files:

api/incidents.php, pages/store-incidents.php

Suggested fix: Add CSRF token to the page and send it via X-CSRF-Token (or _csrf_token in JSON body) on update request.

High Meeting attendee IDs are not scope-validated before insert

Issue: API validates meeting store ownership but inserts attendee IDs directly from client payload.

Risk: A crafted request can associate unintended users to meetings (cross-tenant or out-of-location integrity issue).

Files:

api/meetings.php

Suggested fix: Validate each attendee ID belongs to the same company and permitted location before meeting_attendees insert.

High FLHA other_workers IDs are inserted without tenant checks

Issue: FLHA API accepts worker IDs from payload and writes them directly to flha_workers.

Risk: Forged requests can link out-of-scope users to FLHA records.

Files:

api/flha.php

Suggested fix: Validate submitted worker IDs against company scope (store-based or job-based path) before insertion.

High Unsafe fallback to company_id = 1

Issue: Several files default to tenant/company 1 when company resolution fails.

Risk: Can expose/modify wrong-tenant data instead of failing closed.

Files:

api/checklists.php, api/equipment.php, api/training.php, pages/store-reports.php

Suggested fix: Remove fallback-to-1 behavior and return 403/session error when company context is missing.

Medium Authorization model inconsistencies across modules/pages

Issue: Dashboard uses permission-driven module visibility while many pages use hardcoded role arrays (including role-name drift like JHSC Leader vs JHSC Member).

Risk: Users may see links but be blocked, or role behavior may diverge by page.

Files:

pages/dashboard.php, includes/permissions.php, pages/meetings-list.php, pages/host-meeting.php, pages/training-matrix.php, pages/store-incidents.php

Suggested fix: Centralize route checks via permission helper (e.g., can_access_module()) and normalize role names/constants.

Medium SQL internals exposed in API responses

Issue: Raw DB errors are returned to clients.

Risk: Leaks schema/internal details useful for reconnaissance.

Files:

api/equipment.php

Suggested fix: Return generic errors to clients; log detailed SQL errors server-side only.

Medium No explicit deny rule for .env / dotfiles in root .htaccess

Issue: Root rewrite/security config does not explicitly block direct access to sensitive dotfiles.

Risk: If server allows dotfile access, secrets may be exposed.

Files:

.htaccess, .env

Suggested fix: Add deny rules for .env and dotfiles (except .well-known if needed), or move .env outside web root.

Suggested Fix Order

  1. Restore critical functionality first: fix CSRF mismatch in incident review update flow.
  2. Close direct data-integrity vectors: validate meeting attendee IDs and FLHA worker IDs server-side.
  3. Remove tenant fallback risk: eliminate company_id = 1 fallback and fail closed on missing context.
  4. Harden secrets and response exposure: block .env/dotfiles and stop returning raw SQL errors to clients.
  5. Unify authorization logic: migrate page gates to centralized permission checks and standardize role naming.
  6. Add regression tests: verify CSRF-protected writes, tenant isolation for IDs, and permission-route consistency.
